15-445-lab2:Buffer Pool Manager

Posted on Jan 1, 0001

这个项目的目标是为BusTubDBMS构建一个面向磁盘的存储引擎。存储引擎假设数据库的基本存储位置位于磁盘上。

第一个实验是的我们的存储引擎中实现一个buffer pool(也可以叫buffer cache)。这个缓冲池负责在主存和磁盘之间来回移动物理页。它允许DBMS支持大于系统可用内存量的数据库。缓冲池的操作对于系统的其他部分是透明的。例如，系统使用它的唯一标识符page_id_t向缓冲池申请一页物理页使用并且它不知道这页物理页是否已经存在于内存中或者系统是否需要去磁盘检索这页物理页。

我们的实现必须是线程安全的。多个线程可能会同时访问内部数据结构，因此，我们需要确保使用latches(在操作系统中叫做locks)保护临界区。

我们需要在存储引擎实现以下两个组件：

  • LRU Replacement Policy
  • Buffer Pool Manager

TASK #1 - LRU REPLACEMENT POLICY

这个组件负责跟踪缓冲池中页面的使用情况。我们需要实现一个叫做LRUReplacer的新子类，它位于src/include/buffer/lru_replacer.h，相应的实现位于src/buffer/lru_replacer.cpp中。LRUReplacer继承于抽象Replacer类(src/include/buffer/replacer.h)，包含了指定的函数。

LRUReplacer的大小和缓冲池一样因为它为BufferPoolManager中的所有frame提供了占位符。然而，并非所有的frames都被视为LRUReplacer中的frame。LRUReplacer初始化时没有frame。然后，只有新的unpinned的那些才会被认为在LRUReplacer中。

我们需要实现在课堂上讨论的LRU策略，需要实现下面的方法：

  • Victim(T*): 移除相对于所有由LRUReplacer跟踪的最近最少访问的那些元素，在输出参数中存储它的内容并且返回True。如果Replacer为空则返回False。

  • Pin(T): 这个方法应该在BufferPoolManager中一个页面被固定到一个frame后调用。它应该从LRUReplacer中移除包含着pinned页的帧。

  • Unpin(T): 这个方法应该在pin_count为0时调用。这个方法向LRUReplacer中添加包含着unpinned页的frame。

  • Size(): 这个方法返回当前处于LRUReplacer中的frame数目。

我们可以自由选择STL容器，我们可以假设不会超过内存但必须确保操作是线程安全的。

Implementation Details

  • 什么是frame，什么是page？它们有什么关系？
    内存区域可以看成是一个固定大小的page数组。我们把数组的每一项都叫做frame：页框。
    而磁盘上的被划分为固定大小单元来组织，这个单元叫做page：页面。
    当DBMS请求磁盘上的一个页面时，这个page就会被精确拷贝到一个frame中。具体拷贝到哪个frame视情况而定，而不是按照顺序。page id为1的page不一定拷贝到frame id为1的frame。

Implementation Issue

整体思路就是就是双链表 + hash.KV对就是<frame_id_t, DLinkedNode>，链表结点中存储的值是key。也就是说，key和value是相等的，不过为了换取更高的访问速度，所以选择用双链表结点进行封装。

但是实现时还是会有一些问题：

  • frame的unpinned和pinned状态代表什么？

一个frame状态为unpinned代表这个frame目前没有被任何一个thread所读写，pinned代表这个frame目前至少被一个thread所读写。

  • 为什么LRUReplacer提供的接口和OJ上的接口区别那么大？

LRUReplacer追踪的是最近被访问过的frame。数据库中的LRU和OJ上的LRU所要考虑的有些不同，OJ上的LRU提供了两个接口ValueType get(KeyType key)和void put(KeyType key, ValueType value)。 为什么项目中没有这两个接口？虽然我们有Victim(T*)，能拿到最近最少访问的frame并从LRU中evict掉，为什么不能get到其他frame？这是因为在数据库中我们需要考虑Temporal Locality，相对于其他 frame，最近最少访问的frame将来很大概率不会被再次访问，所以我们可以evict掉。而其他frame因为Temporal Locality可能很快会被再次访问。

  • 为什么Pin(T)要从LRUReplacer中移除pinned状态的frame？

当一个frame为pinned时，说明这页frame上的page正在被某一个线程所读写，此时它的pin_count一定大于0，我们需要在线程结束使用它后，将这页frame加入到LRU中，因为它刚刚被某一线程所读写。

但是我们无法在Pin(T)函数中判断这页frame是否已经结束使用，所以我们得先把它从LRUReplacer中移除，随后frame结束使用的时候一定会调用Unpin(T)，这时候我们再把它加入到LRUReplacer中。 这就是为什么我们还需要Unpin(T)函数的原因。这相当于OJ上LRU的put(KeyType key, ValueType value)中key已经存在的情况，而我们需要对value进行一个update操作，这个操作就相当于(1)在LRU中把这 个元素移除，然后update它的value，(2)更新结束(线程不再读写这个结点)，然后加入到LRU中。OJ上的LRU并不是thread-safe的，单线程说明我们可以直接在LRU中更新对应结点的值，只在put()中就可以单独实现。 但在实际数据库系统中不可以这样，因为会涉及到多个线程并发读写某一页frame的情况，如果写在一个put中，首先会增加了耦合性，LRU和其他模块应该尽可能地独立，其次也不好保护临界区地访问。

所以说，将put()拆成Pin()和Unpin()其实是一个十分良好的设计。

  • 在Unpin(T)中为什么LRU中已经存在frame时，什么都不用做直接返回，而不是添加到LRU尾部，表示最近被访问？

因为在LRU中的frame表示之前已经被线程访问过了，并且到目前操作为止都没有被线程访问过，因为如果需要被线程读写，一定会被evict掉。所以frame没有最近被再次访问，更不用说什么添加到链表尾部。

内存泄漏

我们知道在CPP中手动管理内存是十分危险的，如果使用了new关键字，一定需要和一个delete关键字匹配，如果没有可能会发生内存泄漏。

在Unpin()函数中，我们需要向LRURepalcer添加一个结点，会涉及到new关键字，那么什么时候需要delete呢？就是我们在调用Pin()函数的时候，我们需要在LRUReplacer中移除一个结点的时候使用delete关键字 释放不用的内存。

改进 使用智能指针而不是原始指针可以避免这种pitfalls。

  // before unpin the frame, checkout the pin count first.
