15-445-lab2:B+ Tree

Posted on May 18, 2021

本次编程项目的是在我们的数据库系统中实现索引。索引负责快速检索数据，有了索引，我们就不必在数据库表中扫描每一行，索引为有序记录的快速随机查找和高校访问提供了保证。

我们会实现B+树动态索引。B+树是一种自平衡树，它数据结构分为内部结点和叶子节点,叶子节点包含实际的数据项，而内部节点用来指导搜索，即提供一个路由的作用，指导我们应该前往哪个叶子节点搜索我们想要的数据项。

因为树结构会动态的增长和收缩，我们需要处理逻辑上的分裂和合并

TASK #4 - CONCURRENT INDEX

我们需要更新原始的单线程B+树索引以至于我们可以支持并发操作。我们会使用latch crabbing technique(蟹行技术)，正在遍历索引的线程会在B+树页面上先获取锁然后释放它。一个线程仅在孩子页面被认为是安全的情况下才会释放父页面上的锁。安全的定义在不同的执行操作中是不同的。

  • Search: 从根节点开始，一旦到达孩子节点，拿到读锁之后就可以释放父节点上的读锁。

  • Insert: 从根节点开始，获取写锁，到达孩子节点，获取孩子节点上的写锁，如果孩子节点是安全的，这里安全指的是插入后不会分裂，那么才可以释放父节点上的写锁。如果孩子节点不安全就释放父节点上的锁，可能会造成两个线程同时对父节点进行写操作。

  • Delete: 从根节点开始，获取写锁，到达孩子节点，获取孩子节点上的写锁，如果孩子节点是安全的，这里安全指的是删除后不会造成redistribution和coelesc操作，也就是说至少超过半满，才可以释放父节点上的写锁。

REQUIREMENTS AND HINTS

  • 我们不允许使用全局锁来保护数据结构。换句话说，我们不应该锁住整个索引并且在完成操作的时候释放锁。
  • 助教已经提供了read-write latch(src/include/common/rwlatch.h)的实现，并且已经在page头文件(src/include/storage/page/page.h)下添加了申请和释放锁的辅助函数。
  • 助教不会在B+树索引中添加任何强制性接口。我们可以在实现中编写任何私有/辅助函数，只要保留所有原始的公共接口完整无损以进行测试即可。
  • 对于这个任务点，我们必须使用传入的称为transaction(src/include/concurrency/transaction.h)的参数指针。它提供了当遍历B+树时存储已经获得到锁的页面的函数以及存储在Remove操作期间已经删除的页面的函数。 助教的建议是仔细阅读B+树中的FindLeafPage函数，我们可能想要修改先前的实现(我们可能需要修改这个函数的返回值)以及在这个指定的函数中添加latch crabbing的逻辑。
  • 在缓冲池中FetchPage()函数的返回值是一个指向Page(stc/include/storage/page/page.h)实例的指针。我们可以在Page上申请一个锁，但不能在B+树节点上申请锁。

COMMON PITFALLS

  • 测试程序不会测试线程安全的叶节点扫描操作(并发的迭代器操作)。但是我们应该在不能申请到锁时抛出一个异常以避免死锁。
  • 仔细思考page类中的Unlock()方法和buffer pool manager类中的UnpinPage(page_id, is_dirty)方法之间的顺序和关系。在unpin同样的页面之前必须释放掉获取的锁。
  • 如果正确地实现了并发B+树索引，那么每个线程总是可以获取从根节点到底部叶子节点的锁。当释放锁时，必须按获取锁的同样的顺序(从根节点到底部)释放锁，这样才能避免死锁。
  • 一个需要注意的边界情况是当插入和删除时，成员变量root_page_id也可能会被更新。我们需要负责保护它防止并发更新。(提示：可以添加一个抽象层，使用std::mutex来保护变量)

Implementation

提一下实现过程中遇到的难点以及是如何解决的。

  1. 如何保证根节点不被并发写是安全的？当前实现的代码如下：

    INDEX_TEMPLATE_ARGUMENTS
bool BPLUSTREE_TYPE::Insert(const KeyType &key, const ValueType &value, Transaction *transaction) {
  bool ret = true;
  if (IsEmpty()) {
      StartNewTree(key, value);
      return true;
  }
  ret = InsertIntoLeaf(key, value, transaction);
  return ret;
}

    思考一下，当前B+树还没有创建，这是有两个线程同时执行了Insert操作，这时候就会同时创建一个新的B+树，这显然是我们不希望看到的，我们希望看到的是两个线程的插入操作都会成功，B+树中有两个key，而不是一个key。

    我们显然不像能这样在判断分支之前加锁：

    std::lock_gaurd<std::mutex> gd{mutex_};
if (IsEmpty()) {

    因为这样会导致整个B+树被锁住，每次只能允许一个线程进行插入操作，这样效率很极低，我们只希望多个线程不要同时对同一个叶子节点进行写操作。

    我们需要一个双重检查来保证不能两个线程同时执行StartNewTree函数：

    if (IsEmpty()) {
    std::lock_gaurd<std::mutex> lg{mutex_};
    if (IsEmpty()) {
        StartNewTree(key, value);
        return true;
    }
}
ret = InsertIntoLeaf(key, value, transaction);

  2. 我们解决了两个线程同时创建B+树的问题，但并没有保护根节点标识符root_page_id的并发访问，我的实现中判断B+树是否为空的函数实现如下：

    INDEX_TEMPLATE_ARGUMENTS
bool BPLUSTREE_TYPE::IsEmpty() const { return root_page_id_ == INVALID_PAGE_ID; }

    思考一下如下情形：此时有两个线程t1和t2, t1向B+树插入key：1， t2向B+树插入key：2。t1从磁盘获取一个新的页面，更新了根节点的id，但还没有插入key：1，而此时t2先访问了根节点页面，拿到了写锁，并插入key: 2，此时t1就无法插入key：1了，所以只有在t2完成插入后t1才能拿到根节点上的写锁，插入key：1，这样显然违反了串行化。这里先提供一种方法：利用一个单独的根节点锁来保护根节点，防止根节点分裂而造成的根节点更换。最后操作完成后再释放根节点锁。

  3. 其实问题2中，有更好的解决方案而不需要使用额外的根节点锁，那就是使用虚拟根节点页面(哑根节点页面)。这样子根节点的行为就与所有的孩子节点行为一致，又保护了根节点，防止多个节点同时对根节点进行操作。具体的操作如下：每次插入操作时，先锁住虚拟根节点页面，加入到事务的页面集合中，需要提一下的是，虚拟根节点页面其实是一个Page类的实例，所以我们可以直接利用Page类中提供的读写锁方法，然后再根据key用latch crabbing寻找目标叶子节点。最后还需要提一下，当返回时仍需判断页面是否为空，如果为空说明B+树为空，仍需要取得根节点上的锁，进行创建B+树, 在调用创建B+树的函数前对虚拟根节点上锁，创建后释放虚拟根节点页面上的锁，这样可以保证永远先创建B+树，再插入节点，而不会造成在插入节点的时候发现B+树为空的情况，因为我们先判断的if (IsEmpty()).这样能保证创建B+树的线程先获取虚拟根节点上的锁。

  4. 虽然latch crabbing技术可以保证B+树索引支持并发访问，但是并不是一个高效的选择，因为B+树的节点通常很大，发生分裂、合并的频率较小，其实可以采用乐观加锁的机制，从虚拟根节点开始，对于叶子节点获取正确的锁(写锁)，而父节点只获取读锁，如果目标叶子节点会发生分裂或者合并，即不是安全的，说明判断错误，需要重新走一遍latch crabbing，这样基于乐观加锁的机制可以较好的改善性能问题。

  5. 最后提一下，C++11中的std::move和std::move_backward函数，它们可以用来**“优雅的”**移动容器元素, 但是函数定义的第三个参数result，不能位于前两个参数的区间之内。

    e.g. 在数组中插入一个元素:

    int array[5] = {1, 2, 3, 4, 5};
int n = sizeof(array) / sizeof (int);
int insert_idx = 2;
int val = 100;
std::move_backward(array + insert_idx, array + n, array + n + 1);
array[insert_idx] = val;

    e.g. 在数组中删除一个元素：

    int array[5] = {1, 2, 3, 4, 5};
int n = sizeof(array) / sizeof(int);
int remove_idx = 2;
std::move(array + remove_idx + 1, array + n, array + remove_idx);
resize(n - 1); // 一个伪操作，数组需要手动将size - 1
